# ics-timezone-fixer

Скрипт на Python для подготовки рабочего календаря Outlook к импорту в Google Calendar.

## Зачем

Outlook выгружает события с **Windows-названиями таймзон** в поле `TZID`
(например `TZID=Central European Standard Time`). Google Calendar такие названия
не распознаёт и показывает время неверно. Также в рабочем календаре часто есть
«мусорные» встречи (автосинки, стендапы), которые не нужны в личном календаре.

Скрипт по правилам из `config.json`:

- **удаляет** ненужные события;
- **подменяет строку `TZID`** у нужных событий на распознаваемую Google — при этом
  **цифры времени не меняются** (15:00 остаётся 15:00, переписывается только ярлык
  таймзоны, без пересчёта момента);
- применяет **глобальный словарь замен** `tzid_map` сразу ко всем событиям и к
  соответствующим блокам `VTIMEZONE`.

## Установка

```bash
pip install -r requirements.txt
```

Зависимости: [`requests`](https://pypi.org/project/requests/) (скачивание `.ics`) и
[`icalendar`](https://pypi.org/project/icalendar/) (парсинг/правка). Требуется Python 3.9+.

## Запуск

```bash
# Источник из config.json или переменной окружения:
python ics_timezone_fixer.py --config config.json

# Источник из переменной окружения (удобно для CI/секретов):
ICS_SOURCE_URL="https://outlook.office365.com/.../calendar.ics" \
    python ics_timezone_fixer.py

# Явно указать URL и файл результата:
python ics_timezone_fixer.py --url https://.../calendar.ics --output out.ics
```

Приоритет источника `.ics`: `--url` → переменная окружения `ICS_SOURCE_URL` →
`source_url` из `config.json`. Параметр `--url` также принимает локальный путь или
`file://` URL (удобно для отладки). Уровень логов — `--log-level DEBUG|INFO|WARNING|ERROR`.

## Конфигурация (`config.json`)

```json
{
  "output_file": "processed_calendar.ics",
  "source_url": "",
  "tzid_map": {
    "Central European Standard Time": "Central Europe Standard Time",
    "Russian Standard Time": "Europe/Moscow"
  },
  "rules": [
    { "match": "*Синхронизация*", "action": "delete" },
    {
      "match": "Daily Standup*",
      "action": "change_timezone",
      "parameters": { "new_tz": "Europe/Belgrade" }
    }
  ]
}
```

| Поле          | Назначение                                                                 |
| ------------- | -------------------------------------------------------------------------- |
| `output_file` | Куда сохранить результат (по умолчанию `processed_calendar.ics`).           |
| `source_url`  | URL исходного `.ics`. Можно оставить пустым и задать через `--url`/env.     |
| `tzid_map`    | Словарь `"старое имя TZID": "новое имя TZID"`, применяется ко всем событиям. |
| `rules`       | Список правил, применяемых к событиям по совпадению `SUMMARY`.              |

### Правила (`rules`)

- `match` — маска имени события (`SUMMARY`) с поддержкой wildcards `*` и `?`
  (модуль `fnmatch`). Сопоставление **регистронезависимое**.
- `action`:
  - `delete` — событие полностью исключается из итогового файла;
  - `change_timezone` — у полей `DTSTART`/`DTEND`/`RECURRENCE-ID` строка `TZID`
    заменяется на `parameters.new_tz`. Цифры времени сохраняются.

Правила применяются по порядку. Первое сработавшее `delete` удаляет событие и
прерывает дальнейшую обработку этого события; `change_timezone` применяется и не
прерывает цепочку. События на весь день (DATE) и время в UTC (`...Z`) при смене
TZID пропускаются — у них нет таймзоны-ярлыка.

### `tzid_map` vs `change_timezone`

- `tzid_map` — массовая починка имён таймзон у **всех** событий; дополнительно
  переименовывает блоки `VTIMEZONE`, поэтому файл остаётся самосогласованным
  (Google получает и имя, и правила перехода на летнее время из Outlook).
- `change_timezone` — точечно для событий по маске. Блок `VTIMEZONE` при этом не
  трогается, поэтому `new_tz` должен быть именем, которое Google понимает
  самостоятельно (IANA, например `Europe/Belgrade`), либо покрываться `tzid_map`.

## Логирование

На уровне `INFO` выводится URL источника и итог: сколько событий обработано,
удалено, у скольких изменён TZID и сколько блоков `VTIMEZONE` переименовано.
Уровень `DEBUG` показывает каждое сработавшее правило с именем события.

При ошибках (нет конфига, битый JSON, ошибка скачивания, некорректный ICS) скрипт
завершается с ненулевым кодом и понятным сообщением.

## Запуск по расписанию

Скрипт читает источник из переменной окружения `ICS_SOURCE_URL`, поэтому его удобно
запускать из cron или GitHub Actions, храня URL календаря в секретах. Итоговый
`.ics` сохраняется локально для последующей публикации.
